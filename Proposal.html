ISO Detective

A noir investigation into the test techniques that weren't run.

Course · ITOISQS Standard · ISO/IEC 29119-4 : 2021 Format · Browser web game Session · 10–15 min

1. Game overview

Title, genre, platform, audience

ISO Detective is a single-player, point-and-click educational mystery game delivered as a self-contained browser application (HTML/CSS/JS, no install). Target audience: undergraduate software engineering and quality assurance students approaching ISO/IEC 29119-4 for the first time. Designed for live classroom delivery — projected by the instructor or played individually on student laptops or tablets.

Learning objectives (measurable, ISO 29119-4-aligned)

After one full session a student is able to:
  1. Classify any ISO/IEC 29119-4 technique as Black-Box, White-Box, or Experience-Based with ≥85% accuracy.
  2. Select the correct technique for a described testing scenario (e.g. defined input ranges → EP + BVA).
  3. Explain why omitting a specific technique causes a specific category of defect.
  4. Distinguish between three frequently confused pairs: EP vs BVA, Decision Table vs State Transition, Exploratory vs Error Guessing.
  5. Map a real-world software failure to the missing technique(s) that would have prevented it.

Game concept

The player is a detective summoned to the scene of a software failure. Each "case" is a fictional company whose product caused real harm — wrong fund transfers at a bank, dangerous medication doses at a hospital, miscalibrated GPS in commercial aviation. The detective doesn't read about the bug; they walk the scene, find evidence, and reason backwards to the missing test technique. The game's atmosphere is medium noir: rain on the window, a single amber lamp, classified folders, polaroid evidence, a cork suspect-board. The pedagogy is hidden inside the genre — you are not "studying ISO 29119-4," you are solving cases whose every clue is one ISO clause.

Three cases of escalating difficulty cover the full ISO/IEC 29119-4 technique catalogue: Black-Box (EP, BVA, Decision Table, State Transition, Cause-Effect Graphing, Pairwise, Classification Tree), White-Box (Statement, Branch, Condition, MC/DC, Path), and Experience-Based (Error Guessing, Exploratory, Checklist-Based). A persistent in-game Field Guide contains every clause, definition, and worked example. A Verdict screen at the end of each case replays every wrong link as a one-paragraph mini-lesson.

Mechanics & Pedagogy

2. How the game teaches

The evidence → consequence → technique loop

Every hotspot in a scene encodes one ISO clause. The hotspot is a recognisable artefact (a test plan, a whiteboard, a sticky note, a code printout) showing what was done. The evidence card it produces shows the consequence — not in abstract terms but in the loss it caused: 847 fraudulent transfers, an accepted -5mg dose, a flight diverted at 179.9999° longitude. The student is then asked to link the evidence to the formal technique that would have prevented it. This forces the student to reason from observable harm back to the underlying clause — the same direction practising testers must reason in post-mortems.

Wrong-answer system as the primary teaching mechanism

Every hotspot has both a correct technique and at least one plausible-but-incorrect trap. When a student picks a trap, the game does not shame; it teaches. A noir-styled "Wrong lead, Detective" panel opens, side-by-side comparing the chosen technique and the correct technique, each with its ISO clause reference. The student then tries again at reduced score. This converts wrong answers into compare-and-contrast learning events — the deepest moments of the game, by design.

"Wrong lead, Detective. You picked Branch Testing. The evidence points to Condition Testing — here's why."

Rule outline

  • Each case opens with a folder slamming onto the desk; the player is briefed in 30 seconds.
  • Investigate the scene by clicking 8–12 amber-pulsing hotspots; each yields one polaroid evidence card.
  • On the suspect-board, drag (click-link) each evidence card to one of the case's candidate techniques.
  • The Field Guide is always one tap away — the game rewards understanding, not memorisation.
  • After all three cases, the Verdict screen issues a final detective rank and replays every cold case.

Extensibility

Cases live in a single JSON-shaped data module: id, title, severity, summary, hotspot list (position, evidence body, ISO clause, correct technique, trap techniques), and the Field Guide entries are shared. A future cohort can author a fourth case (e.g. The TrainSim Signal Mishap) by adding ≈120 lines to that module — no new screens, animations, or styling required. Suggested student exercise: "Author one hotspot end-to-end" — pick a real CVE, write the evidence, identify the missing ISO 29119-4 technique, and submit a pull request to the cases module.

Definition of done (delivered)

  • 3 cases × 8–10 hotspots = 26 evidence/technique linkages
  • 5 screens: Landing, Case File, Investigation, Verdict, Teacher Dashboard
  • Full Field Guide with all 15 techniques
  • Wrong-answer system with ISO-accurate trap explanations
  • Tweaks panel (animations, audio, timer, hints)
